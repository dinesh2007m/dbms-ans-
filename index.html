(i)

A legacy payroll system follows this sequence: data entry → manual verification → file storage → report generation.

After migrating to a database system, several steps are abstracted by the DBMS.

 Re-map the process flow by identifying which steps are absorbed by the database system and explain how data independence alters the system state transitions.

Answer:

Re-mapped Process using DBMS:

1.     Data Entry → DBMS Input Interface

o   Data is entered and stored directly into the database tables using DBMS forms or applications.

2.     Manual Verification → DBMS Integrity Constraints

o   Verification is handled automatically using constraints, validation rules, and triggers.

3.     File Storage → Database Storage Management

o   DBMS manages data storage, indexing, and data organization internally.

4.     Report Generation → Query Processing

o   Reports are generated using SQL queries and reporting tools.

Effect of Data Independence:

·        Data independence allows changes in database structure without affecting applications.

·        It ensures smooth system state transitions even if schema or storage structure changes.

Keywords: DBMS abstraction, integrity constraints, SQL queries, data independence, storage management.

1B-q1

(i)

A simplified system environment is described as follows: end users interact with application programs; application programs directly access stored data files; data is physically stored on disk.

 Compared to a standard database system environment, one critical architectural layer is absent.

Identify this missing layer, explain its functional role, and analyze how its absence impacts data abstraction, security enforcement, and long-term maintainability.

Answer:

1. Missing Architectural Layer:

The missing layer is the Database Management System (DBMS) layer.
2. Functional Role of DBMS:

DBMS acts as an intermediate layer between application programs and the database.
It provides data management, query processing, security control, and data abstraction.
3. Impact of Absence:

Data Abstraction:
Without DBMS, applications directly access files, so no abstraction layers (physical, logical, view levels) exist.
Changes in file structure require modification of application programs.
Security Enforcement:
File-based systems provide limited security.
No centralized access control or authorization mechanisms.
Long-term Maintainability:
Programs become tightly coupled with data files.
System maintenance becomes complex and time-consuming when data structures change.
Keywords: DBMS layer, data abstraction, access control, file-based system, maintainability.

1C-q1

(i)

A traditional relational database organizes data using tables, rows, and fixed schemas. A

NoSQL document database stores data as flexible JSON-like documents.

 Using a real-world analogy (such as filing systems, warehouses, or libraries), translate how data organization, schema flexibility, and query behavior differ between these two models. Your analogy must clearly justify why NoSQL systems are better suited for certain modern applications.

Answer:

Analogy: Library System

1. Relational Database (RDBMS):

Similar to a library catalog system with fixed forms.
Every book record must follow a strict structure (title, author, publisher, year).
Data is organized in tables with rows and columns.
Queries use structured SQL to retrieve related data from multiple tables.
2. NoSQL Document Database:

Similar to a digital library where each book record can have different details.
Information is stored as flexible JSON-like documents.
Each document can have different fields depending on the data type.
3. Schema Flexibility:

Relational DB: Uses a fixed schema, changes require schema modification.
NoSQL: Uses a dynamic schema, allowing easy addition of new fields.
4. Query Behavior:

Relational DB: Queries often require joins between tables.
NoSQL: Data is stored together, enabling faster retrieval without complex joins.
5. Why NoSQL is Suitable for Modern Applications:

Supports large-scale data, flexible structure, and fast access, which is useful for web applications, social media platforms, and big data systems.
Keywords: Relational database, NoSQL, fixed schema, dynamic schema, JSON documents, scalability.

1D-q1

(i)

A transaction in a DBMS follows this high-level sequence:

Transaction start → Read data → Modify data → Commit

 During execution, a system crash occurs immediately after data modification but before commit.

Reconstruct the internal DBMS sequence of actions involving relevant components that ensures database consistency after recovery, and explain how state transitions are managed.

Answer:

1. Write-Ahead Logging (WAL):

Before modifying data, the DBMS log manager records the update in the log file.
2. Crash Occurs Before Commit:

Since commit record is not written, the transaction is considered incomplete (uncommitted state).
3. Recovery Process:

During restart, the recovery manager performs UNDO operation using the log records, restoring the database to the previous consistent state.
State Transition:

Transaction moves from Active → Failed → Aborted, ensuring database consistency.
Keywords: Write-Ahead Logging, recovery manager, UNDO operation, transaction states, database consistency.

A DBMS handles hundreds of concurrent transactions. Two system parameters are varied independently:

•  Size of the buffer pool

•  Frequency of transaction commits

 Analyze which parameter has a more dominant impact on throughput under high concurrency and justify your reasoning by linking it to internal DBMS component behavior rather than surface-level performance metrics.

Answer:

Dominant Parameter:

Size of the Buffer Pool
Reasoning:

Buffer Manager Role:
The buffer pool stores frequently accessed data pages in main memory, reducing disk I/O operations.
Impact on High Concurrency:
A larger buffer pool allows multiple transactions to access data from memory, increasing throughput and reducing access latency.
Commit Frequency Impact:
Commit operations mainly affect the log manager and transaction manager, but they occur only at transaction completion and have less impact on continuous data access.
Keywords: Buffer pool, buffer manager, disk I/O, concurrency, transaction throughput.

1E-q1

(i)

A database designer models a university system using a single relation:

STUDENT_ALL(StudentID, Name, Department, CourseID, CourseName, FacultyName). During operation, the following problems appear: repeated updates, inconsistent faculty names, and inability to delete a course without losing student information.

 Identify the deep conceptual flaws in this design, explain the underlying relational principles being violated, and outline how restructuring the relations resolves these issues.

Answer:

1. Conceptual Flaws in the Design:

The relation stores student, course, and faculty information in a single table, causing data redundancy.
Multiple students taking the same course lead to repeated course and faculty data.
2. Violated Relational Principles:

Normalization Principle (1NF/2NF/3NF):
Non-key attributes like CourseName and FacultyName depend on CourseID, not directly on StudentID.
Update Anomaly:
Changing a faculty name requires updating multiple rows.
Deletion Anomaly:
Deleting a course record may remove student information unintentionally.
Insertion Anomaly:
Cannot insert course details without a student record.
3. Restructuring the Relations (Normalization):

STUDENT(StudentID, Name, Department)
COURSE(CourseID, CourseName, FacultyName)
ENROLLMENT(StudentID, CourseID)
4. How This Resolves the Issues:

Eliminates data redundancy.
Maintains data consistency and integrity.
Supports independent updates and deletions.
Keywords: Normalization, functional dependency, update anomaly, deletion anomaly, relational schema.

(i)

A hospital wants to represent patients, doctors, and appointments.

Patients have multiple appointments, doctors attend many patients, and each appointment must involve exactly one patient and one doctor, with a scheduled time.

 Construct a relational model starting from this real-world description. Clearly justify the choice of relations, primary keys, and foreign keys, and explain how the model preserves real-world semantics.

Answer:

1. Relations (Tables):

PATIENT(PatientID, PatientName, Age, Contact)
DOCTOR(DoctorID, DoctorName, Specialization)
APPOINTMENT(AppointmentID, PatientID, DoctorID, ScheduledTime)
2. Primary Keys:

PatientID → Primary Key in PATIENT
DoctorID → Primary Key in DOCTOR
AppointmentID → Primary Key in APPOINTMENT
3. Foreign Keys:

PatientID in APPOINTMENT → references PATIENT(PatientID)
DoctorID in APPOINTMENT → references DOCTOR(DoctorID)
4. Preservation of Real-World Semantics:

One patient can have multiple appointments → represented through PatientID in APPOINTMENT.
One doctor can treat many patients → represented through DoctorID in APPOINTMENT.
Each appointment connects exactly one patient and one doctor at a scheduled time, maintaining relationship integrity.
Keywords: Primary key, foreign key, relational model, referential integrity, one-to-many relationship.

1F-q1

(i)

 A development team struggles to understand why object-oriented classes with inheritance and methods do not map cleanly to relational tables.

 Using a real-world analogy (such as organizational hierarchies, blueprints vs. buildings, or contracts vs. records), explain the object–relational impedance mismatch. Then conceptually position ORM as a translation layer and justify which mismatches it resolves and which it fundamentally cannot.

Answer:

1. Real-World Analogy (Blueprint vs Building):

Object-Oriented Model: Similar to a blueprint, where objects contain attributes, methods, and inheritance.
Relational Model: Similar to a building record, where information is stored in tables with rows and columns.
2. Object–Relational Impedance Mismatch:

Objects: Store data + behavior (methods) and support inheritance.
Relational Tables: Store only data, with no behavior or inheritance support.
This structural difference creates a mapping mismatch.
3. Role of ORM (Object Relational Mapping):

ORM acts as a translation layer between object-oriented programs and relational databases.
It converts objects into database tables and rows automatically.
4. Mismatches ORM Can Resolve:

Mapping class attributes to table columns.
Handling object persistence and data retrieval.
Simplifying database operations through object methods.
5. Mismatches ORM Cannot Fully Resolve:

Differences in inheritance representation.
Complexity of object relationships vs relational joins.
Performance issues caused by object-to-table translation.
Keywords: Object-relational impedance mismatch, ORM, inheritance mapping, relational tables, object persistence.

1F-q2

(i)

A public-sector payroll system is evaluated for compliance before nationwide rollout.

Three deployment options are proposed:

 Option A: A legacy relational database that partially implements relational features

 Option B: A modern relational DBMS that claims compliance with Codd’s principles

 Option C: A hybrid system using relational storage with heavy application-side logic

 The system must ensure data independence, non-bypassable integrity rules, and consistent query behavior across tools.

 Using the intent of Codd’s rules as evaluation criteria, select the most defensible option and justify your choice by identifying which rules become ethically and operationally critical in public data systems, and why partial compliance is insufficient.

Answer:

1. Most Defensible Option:

Option B: A modern relational DBMS compliant with Codd’s rules.
2. Critical Codd’s Rules for Public Systems:

Data Independence (Logical & Physical):
Applications remain unaffected when database structure or storage changes.
Integrity Independence:
Integrity constraints (such as primary keys and referential rules) must be defined and enforced by the DBMS, not by application programs.
Non-bypassable Data Access Rule:
All data access must go through the DBMS query system, ensuring consistent security and validation.
Uniform Data Access (Relational Model):
Users and tools access data through standard query language (SQL) ensuring consistent query behavior.
3. Why Other Options Are Insufficient:

Option A (Legacy System):
Partial implementation may violate key relational principles, causing data inconsistency and weak integrity enforcement.
Option C (Application-side Logic):
Integrity rules can be bypassed if applications fail, reducing data reliability and security.
4. Importance for Public Data Systems:

Government systems require high reliability, transparency, and data integrity, making full relational compliance essential.
Keywords: Codd’s rules, data independence, integrity constraints, relational DBMS, SQL consistency.
